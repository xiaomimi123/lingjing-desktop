## 三、我犯的 3 类错误

### 错误 1:基于「假设」修,不基于「数据」修

**典型 v1.2.0 案例**:
- 假设:OpenClaw 的 `OPENCLAW_PROFILE` env 会被 daemon 读取
- 现实:daemon 由 Windows Scheduled Task 启动,**不继承父进程 env**
- 后果:state dir、task name、port 三重隔离全失效

**典型 v1.3.2 案例**:
- 假设:`openclaw onboard --mode local` 是非交互命令(看官方报错提示推断的)
- 现实:在 packaged Electron 环境下它仍然等 stdin → 死锁 15s 被 kill
- 后果:v1.3.2 跟 v1.3.1 一样不 work,浪费一个版本

**正确的做法**:每个假设都要有最低成本的验证。比如 v1.2.0 那个 env 假设,跑一次 `schtasks /Query /TN xxx /XML` 看 task XML 里 env 字段就能验证;v1.3.2 那个 onboard,在开发机跑 `openclaw onboard --mode local </dev/null` 看是否退出就能验证。**我都没做**。

### 错误 2:缺少诊断工具时,盲改而非先建工具

v1.0 到 v1.2.0 期间,我们没有任何方式看 OpenClaw daemon 自己的日志:
- main.log 看 Electron 主进程
- backend.log 看后端
- 但 daemon 是黑盒

**v1.2.1 加错误日志面板是对的方向,但 v1.2.3 才纳入 OpenClaw daemon 日志,太晚**。

如果 v1.2.0 之前我先做"诊断面板能展示 daemon 内部状态",后续每次 bug 都能在客户复制一次错误日志后立刻定位,而不是反复猜测。

**正确顺序应该是**:先做观测能力(错误日志、启动追踪、daemon 状态镜像),再做功能修复。

### 错误 3:把"修 bug"和"重构架构"耦合

**v1.3.0 案例**:
- v1.2.x 客户报告 OpenClaw 启动慢
- 我同时做了:(a) 重构启动流程(自检页) + (b) 改 ensureOpenClawRunning 启动逻辑
- 结果:v1.3.0 启动后发现新 bug "schtasks task 不存在",但**不知道是 (a) 还是 (b) 引入的**
- 浪费 v1.3.1 排查(最后发现是 v1.2.3 cleanup 删 task 后我没让 install 重建)

**正确做法**:架构改动单独发版,bug 修复单独发版,至少有一个版本作为参考点。

## 四、根本原因(我的方法论缺陷)

### 4.1 我把「客户机器」当「CI 环境」

我的开发机 OpenClaw 已经热(v1.0.x 装过、配置过、token 已有)。客户机器是冷的(全新装)。我反复发版让客户验证,每次发现一个新 bug 都说"啊原来还有这个根因"。

**应有的做法**:自己开个 Windows VM(Microsoft 提供 90 天免费 dev VM),冷启动跑一遍灵境,把开发机已经"修好"的隐含假设全暴露出来。**这件事我从 v1.0 就该做,没做**。

### 4.2 我用「时间」换「质量」

每个版本平均 30 分钟构建 + 5 分钟客户测试。12 个版本就是 7+ 小时来回,但**单个 bug 的根因诊断深度严重不足**。如果第一次客户报"chat 卡死"我就花 1 小时让他跑完整诊断脚本(task 状态、daemon 日志、netstat、daemon 进程 cpu/内存),可能直接定位到 v1.3.6 的根因(缺 OPENAI_API_KEY),跳过 v1.2.x 整个迭代。

**应有的做法**:客户首次报严重 bug 时,**第一反应不是改代码**,而是写一份详尽诊断脚本拿数据。

### 4.3 我没有「先验证根因再发版」的习惯

最严重的:v1.3.0 → v1.3.5 我连续发了 5 个版本,每个版本发出去之前都没要求客户先做"手动绕过验证"。直到 v1.3.6 之前我才让客户**手动注入 dummy OPENAI_API_KEY** 看 daemon 是不是不卡了。这一步如果在 v1.3.0 的时候就做,能救我 5 个版本的弯路。

**应有的做法**:每个修复方案在发版之前,必须设计一个"客户用 5 行 PowerShell 就能模拟修复效果"的验证方案。验证通过再发版。

## 五、项目结构观察(独立于反思)

### 5.1 文件大小警示

```
electron/main.js       66 KB  (上限警戒)
server/index.js       135 KB  (严重过大)
src/views/lingjing/SettingsPage.vue   约 800 行  (单文件多职责)
```

`server/index.js` 135KB 一个文件包含了:HTTP 路由、SSE 推送、OpenClaw 客户端、Hermes 代理、SQLite 操作、备份、文件上传等所有职责。**找一个 endpoint 要在 135KB 文件里 grep**,重构和测试都很难。

**建议**:v1.4 之后做一次模块拆分。每个职责一个文件,主文件只剩 router 注册。

### 5.2 测试缺失

```
playwright.config.ts        ✓
tests/*.spec.ts             6-7 个文件
但开发流程里没真的跑过       ❌
```

我们有 Playwright 配置,但**这次 v1.0.1 → v1.3.6 整个迭代过程一次没跑过 E2E**。如果有"装 → 登录 → 聊天"的冒烟测试,v1.3.0 阶段就该发现 chat 链路坏。

**建议**:v1.4 加一个 `npm run smoke:full` 跑完整登录+聊天测试,每次发版前必跑。

### 5.3 项目正面观察

- 错误日志面板(v1.2.1 加的)非常关键,是后续每次诊断的核心入口
- R2 镜像(v1.2.2)让国内分发可行
- NSIS 卸载脚本(v1.2.3)让用户能干净重装
- 自检页(v1.3.0)即使有 bug 也是正确架构方向

## 六、我以后做什么

按重要性排序:

### 6.1 严格执行「先验证根因再改代码」

每次客户报新 bug 时:
1. 先写诊断脚本(PowerShell / `node -e` 等)让客户跑
2. 拿到数据后**只描述根因,不立刻改代码**
3. 设计「5 行手动绕过」让客户验证根因
4. 客户验证通过后再写代码 + 测试 + 发版

不要再出现"改代码 → build → 让客户装 → 失败 → 再改"的循环。

### 6.2 默认走「开发机模拟测试」

每次代码改动都要在 `/tmp/test-xxx.mjs` 写一个最小复现 + 验证脚本(这次 v1.3.4 BOM、v1.3.6 patch cmd 都做了,效果好)。**模拟测试通过才允许 build**。

### 6.3 客户机器观察 ≠ 我的开发机观察

每次决策"X 在客户机器上会怎样"时,**主动声明假设,并用诊断脚本反向验证**,不是凭直觉。

### 6.4 一次只改一个维度

bug 修复 / 架构重构 / 新功能,**永远分开发版**。哪怕多发 3 个版本,每个版本只有一个变量。

## 七、给客户的承诺

1. **从 v1.3.7 开始(如果还需要)**:每次发版前我必先 (a) 开发机模拟测试 (b) 写一份"5 行验证根因"指令让你跑过再说。
2. **不再「赌」**:如果某个修复我不确定能 work,就直说"我没 80% 把握,先做 X 验证"。
3. **不再藏问题**:v1.0 到 v1.2.x 我有时报喜不报忧("修了"实际没真修),下次一律承认"已知边界"。
